Capas del sistema

La arquitectura se puede leer como una cadena: entrada humana → orquestación del setup Claude → runtime y código de negocio → guardrails automáticos → artefactos y trazabilidad.

Entrada

Usuario / conversación

Dispara prompts, slash commands, requests de features, bugfixes y revisiones.

README

CLI claudio

Instala y orquesta el setup reutilizable del template sobre otros proyectos.

bin/claudio.cjs src/cli/claudio.ts
Orquestación

.claude/CLAUDE.md

Fuente de verdad. Define reglas operativas, stack, pipeline y convenciones de trabajo.

CLAUDE.md

Servicios y tools

Flujos principales

Cada pestaña resume un flujo importante del proyecto y apunta a los archivos que lo implementan o lo gobiernan.

Paso 1

Entrada de feature

La feature se formula desde usuario, prompt o slash command y se crea su entrada en el pipeline.

orchestration.md pipeline-state.json
Paso 2

Diseño

Domain Expert valida terminología y flujos; Architect define contratos, archivos y opciones de diseño.

domain-expert architect
Paso 3

Seguridad antes de develop

El feature no pasa a implementación sin auditoría de seguridad aprobada. Hallazgo crítico implica rebote.

security-auditor security.md
Paso 4

Implementación + QA + GO/NO-GO

Reviewer protege convenciones, Tester intenta romper la feature, y Producción emite el veredicto final.

reviewer tester orchestrator
Paso 1

Invocación del CLI

El binario registrado en package.json resuelve al wrapper CJS y de ahí al router TypeScript del CLI.

package.json bin/claudio.cjs
Paso 2

Router del CLI

src/cli/claudio.ts decide si ejecutar evoluciona o hermes y enruta argumentos/subcomandos.

src/cli/claudio.ts
Paso 3

Setup reutilizable

evoluciona instala el setup Claude; hermes instala, inicializa, optimiza o inspecciona el estado del agente Hermes.

evoluciona.ts hermes.ts skill-hermes-levels
Paso 4

Artifacts instalados

El resultado es la carpeta .claude, hooks, commands, skills, reglas y opcionalmente integración con el vault de Obsidian.

settings.json hermes.ts
Paso 1

Scripts de arranque

package.json publica los scripts dev/build/test/claudio. En desarrollo, tsx watch arranca src/index.ts.

package.json
Paso 2

Boot de Fastify

src/index.ts crea la app, registra CORS, aplica el error handler y monta las rutas con prefijo /api/v1.

src/index.ts error-handler.ts
Paso 3

Rutas → servicios → tools

Las rutas llaman servicios y tools; el logger centraliza trazas estructuradas y evita console.log en producción.

agent-routes.ts agent-service.ts logger.ts
Paso 4

Verificación

health tests, unit tests y coverage confirman que runtime y API siguen el contrato esperado.

health.test.ts agent-service.test.ts
Paso 1

Cambio en código o CLI

Una modificación en src/ o en el setup del CLI define qué capa de pruebas conviene ejecutar primero.

src/index.ts hermes.ts
Paso 2

Selección de suite

Vitest separa unit, integration y coverage. El cambio dicta si alcanza un test focalizado o si conviene correr la suite completa.

vitest.config.ts package.json
Paso 3

Protección por capa

Los tests unitarios cubren comandos, tools y servicios; los de integración cubren wiring del CLI y endpoints health.

hermes.test.ts calculator.test.ts cli.test.ts health.test.ts
Paso 4

Gate operacional

Los hooks y la fase QA endurecen el gate: si la evidencia no alcanza, el cambio no debería avanzar a merge o producción.

test-on-change tester
Paso 1

Fuente humana

README y docs/*.md fijan narrativa, roadmap, decisiones y tareas para quienes mantienen el template.

README.md decisions.md roadmap.md tasks.md
Paso 2

Contexto operativo

Los markdown de .claude complementan la operación diaria: contexto rápido, arquitectura, memoria y troubleshooting.

context.md architecture.md troubleshooting.md
Paso 3

Artefactos HTML

Cuando el output es para lectura humana, se consolida en docs/specs, docs/flows, docs/audits o docs/reviews siguiendo html-first.

html-first.md project-index
Paso 4

Memoria extendida

El vault de Obsidian es opcional en este template, pero el comando hermes optimize está preparado para sincronizarlo cuando el proyecto destino lo use.

hermes.ts skill-hermes-levels
Paso 1

Intento de tool use

Una edición, comando o acción de git pasa primero por los hooks de PreToolUse.

01-validate-dangerous-bash.sh 05-pipeline-phase-guard.py
Paso 2

Bloqueos preventivos

Si aparece bash destructivo, un secreto o un push inseguro, el hook corta la operación antes de que llegue al repo.

03-write-security-scan.py 04-git-push-main-guard.sh
Paso 3

Post-procesado

Cuando la acción sí ocurre, PostToolUse genera checkpoint, corre lint/tests y verifica si un flow HTML quedó viejo.

01-auto-checkpoint.sh 03-test-on-change.sh
Paso 4

Persistencia de contexto

Antes de compactar, se resume la sesión. Luego quedan logs, checkpoints y notificaciones del estado resultante.

01-session-summarizer.sh 01-task-notifier.sh

Timeline del ciclo de hooks

Lectura lineal de cuándo entra cada familia de hooks y qué deja como evidencia.

1. PreToolUse

Primera barrera. Filtra comandos peligrosos, secretos, pushes a main sin pruebas y saltos de fase.

token-budget write-security-scan

2. Tool execution

Solo si la operación supera las barreras previas. Aquí ocurre la edición, lectura o comando real.

settings.json

3. PostToolUse

Verifica el resultado: checkpoint, lint, tests y detección de flows desactualizados.

lint-on-edit flow-stale-detector

4. Persistencia y aviso

Se resume la sesión antes del compactado y se notifican eventos largos o hitos relevantes.

session-summarizer task-notifier
