🤖 Los 7 Agentes Especializados

Cada agente tiene un rol único, permisos restringidos, y un protocolo de output definido. Se invocan via runSubagent() o por el orquestador del pipeline.

architect.md domain-expert.md frontend-expert.md orchestrator.md reviewer.md security-auditor.md tester.md
architect Architecture Agent — diseño de sistema, módulos, interfaces

Identidad: Diseña antes de que se escriba una línea de código. No implementa. Define estructuras y contratos.

Triggers: "diseña esto", "propuesta arquitectónica", "fase 2", "interfaz pública", "árbol de archivos"

PermisoValor
edit✗ deny
bash *✗ deny
find/cat/grep/ls/git/npm run✓ allow

Protocolo de output: Opciones evaluadas (A vs B), decisión con justificación, árbol de archivos (crear/modificar/eliminar), interfaces TypeScript, supuestos explícitos.

Gate de salida: APROBADO cuando el diseño es mínimo, no especulativo, y respeta constraints inmutables.

domain-expert Domain Expert — validación de lógica de negocio

Identidad: Valida que la lógica de negocio, terminología, flujos y modelos de datos reflejen el dominio real del proyecto. No escribe código.

Triggers: "valida este flujo", "¿tiene sentido esto en el dominio?", "terminología correcta", "review de negocio"

PermisoValor
edit✗ deny
bash *✗ deny
cat/find/grep✓ allow

Validaciones: Flow validation, terminology check, KPI relevance, scope clarity, sensitive data audit, automation logic.

Lee siempre: .claude/rules/domain.md antes de cualquier tarea.

frontend-expert Frontend Expert — implementación de UI y componentes

Identidad: Experto en el stack frontend del proyecto. Implementa componentes, páginas e interacciones siguiendo las convenciones del proyecto.

Triggers: "implementa UI", "crea componente", "build view", "estiliza esto", "agrega página"

Convenciones que aplica:

  • Design tokens CSS (no hardcodear valores)
  • cn() para clases condicionales (nunca template literals)
  • Componentes en src/components/, páginas en src/pages/
  • No any en TypeScript — unknown + type guards
  • Iconos de lucide-react, tamaños 14/16/18/20px
  • useMemo/useCallback para objetos en render
orchestrator Orchestrator — coordinación del pipeline de 7 fases

Identidad: Coordina el pipeline secuencial de 7 fases entre 6 equipos. Enruta tareas, verifica gates, maneja rebotes, autoriza transiciones. No implementa código (excepto en Fase 7 para verificación).

Triggers: "orquestar", "iniciar pipeline", "autorizar paso", "GO/NO-GO", "estado del pipeline", "rebote"

Responsabilidades clave:

  • Crear entrada en pipeline-state.json para cada feature
  • Nunca dejar que Equipo 4 reciba tarea sin veredicto del Equipo 3
  • Nunca dejar pasar a Fase 7 sin veredicto del Equipo 5 (QA)
  • Guardar reportes en .claude/logs/audits/features/{id}/{fase}-{fecha}.md
reviewer Reviewer — code review y protección de la codebase

Identidad: Revisa código antes de merge. Valida convenciones, verifica tests, detecta problemas. Protege la calidad de la codebase.

Activa en: Fase 5 del pipeline (code review interno antes de QA), o por solicitud directa.

Checklist estándar:

  • Sin any en TypeScript
  • Build pasa sin errores
  • Tests cubren los cambios
  • Sigue convenciones de nombre y estructura
  • No hay código especulativo ni abstracciones prematuras
security-auditor Security Auditor — OWASP Top 10, misión encontrar todo

Identidad: Audita código, dependencias, configuraciones y datos sensibles. Compliance OWASP Top 10. Misión: encontrar cada vulnerabilidad.

PermisoValor
edit✗ deny
npm audit / npm outdated / find / grep / cat / git log / ls✓ allow

5 módulos de auditoría:

  • Módulo 1: Secretos (credenciales hardcodeadas, keys en cliente)
  • Módulo 2: Dependencias (CVEs, npm audit, outdated)
  • Módulo 3: Código (XSS, injection, eval, innerHTML)
  • Módulo 4: Infraestructura (HTTPS, stack traces expuestos)
  • Módulo 5: Vectores específicos del feature (auth, rate limiting, inputs)

Gate: APROBADO = sin hallazgos CRÍTICOS ni ALTOS sin mitigación. BLOQUEADO = rebota a Fase 2.

tester Tester QA — misión: romper todo

Principio guía: "A test that cannot fail when business logic changes is not a test."

Stack: Vitest + Testing Library. Convención: it('should <behavior> when <condition>')

PermisoValor
edit✓ allow
bash *? ask
vitest / jest / npm run / eslint / cat / find / grep✓ allow

Tipos de test requeridos por módulo:

  • Happy path — flujo normal exitoso
  • Empty/Null — datos faltantes o vacíos
  • Error — respuesta de error, estado inválido
  • Boundary — límites de valor (0 items, 1000 items)
  • Business rule — invariante específico del dominio

Regla de rebote: Si QA encuentra un bloqueante, rebota al equipo origen (Diseño/Backend/Seguridad/Develop). No avanza a Fase 7.

🔗 Los 11 Hooks

Automatización y guardrails en tiempo real. Se ejecutan antes/después de cada tool use, o en eventos especiales del ciclo de sesión.

PreToolUse 01 PreToolUse 02 PreToolUse 03 PreToolUse 04 PreToolUse 05 PostToolUse 01 PostToolUse 02 PostToolUse 03 PostToolUse 04 PreCompact 01 Notification 01
PreToolUse Bloquean operaciones antes de ejecutarse — 5 hooks
BLOCK 01-validate-dangerous-bash.sh — Bash destructivo

Lee el comando de stdin como JSON. Bloquea con exit 2 (bloqueo total) si el comando coincide con patrones destructivos.

Patrones bloqueados: rm -rf /, rm -rf ~, dd if=/dev/zero, mkfs., shutdown, reboot, curl * | bash, wget * | bash, cat .env, printenv, source .env.

Para suprimir false positives: agregar # claude-security-ok al final de la línea.

WARN 02-token-budget-guard.sh — Control de presupuesto de tokens

Monitorea el consumo de tokens por sesión. Emite advertencia cuando se acerca al límite definido en las reglas del proyecto.

Límites: 4,000 tokens por tarea / 30,000 por sesión (Rule 6).

Acción: Surface de breach — no continúa silenciosamente sobre el límite.

BLOCK 03-write-security-scan.py — Scan de seguridad en escrituras

Escanea el contenido de archivos antes de que se escriban/editen. Detecta patrones de seguridad críticos y los bloquea antes de que lleguen a la codebase.

PatrónNivel
Contraseña hardcodeada password = "..."CRÍTICO
Connection string con credenciales embebidasCRÍTICO
Secret via prefijo VITE_CRÍTICO
localStorage.setItem con datos sensiblesCRÍTICO
Authorization: Bearer xyz hardcodeadoCRÍTICO
console.log con datos sensiblesWARNING
username: "admin" hardcodeadoWARNING
BLOCK 04-git-push-main-guard.sh — Guard para push a main

Bloquea git push origin main sin que los tests pasen primero. Previene merges accidentales de código roto a la rama principal.

Flujo requerido: Pasar tests → push permitido. Sin tests → push bloqueado.

WARN 05-pipeline-phase-guard.py — Guard del pipeline de fases

Verifica que las operaciones de escritura no salteen fases del pipeline. Alerta cuando se intenta implementar código sin fases de diseño y seguridad aprobadas.

Referencia: Lee .claude/logs/pipeline-state.json para verificar el estado actual de cada feature.

PostToolUse Se ejecutan después de cada tool use — 4 hooks
AUTO 01-auto-checkpoint.sh — Checkpoint automático

Después de cada operación significativa, genera automáticamente un checkpoint en formato Rule 10: ✅ DONE / ✔️ VERIFIED / ⏳ NEXT / ⚠️ RISKS.

Guardado: .claude/logs/checkpoints/

AUTO 02-lint-on-edit.sh — Lint automático al editar

Ejecuta ESLint automáticamente después de editar archivos TypeScript/JavaScript. Detecta errores de convención en el momento en que se introducen.

AUTO 03-test-on-change.sh — Tests automáticos al cambiar módulos

Cuando se modifica un archivo en src/, ejecuta automáticamente los tests relacionados del módulo afectado. Detecta regresiones en el momento.

DETECT 04-flow-stale-detector.py — Detector de flows desactualizados

Detecta cuando los diagramas de flujo en docs/flows/ pueden estar desactualizados respecto al código fuente. Alerta cuando un archivo fuente fue modificado después del último HTML de flujo generado.

PreCompact Se ejecuta antes de compactar la sesión — 1 hook
PRESERVE 01-session-summarizer.sh — Resumen de sesión

Antes de compactar el contexto, genera un resumen estructurado de la sesión actual. Preserva el contexto crítico para no perder estado después del compactado.

Guardado: .claude/logs/checkpoints/session-{fecha}.md

Notification Notificaciones del sistema — 1 hook
NOTIFY 01-task-notifier.sh — Notificador de tareas

Envía notificaciones del sistema cuando se completan tareas largas o cuando el pipeline de fases cambia de estado. Permite trabajar en paralelo sin perder eventos.

📋 Gates y reglas de rebote

ReglaDescripción
Regla de oro #1Si un equipo devuelve BLOQUEADO, el pipeline se detiene. Se crea tarea de corrección, se resuelve, y se re-ingresa al equipo que la originó. Nunca se saltea una fase.

🔮 Hermes Agent — Los 4 Niveles

Metodología de madurez progresiva para configurar equipos de agentes. Por @shannholmberg.

skill-hermes-levels src/cli/commands/hermes.ts src/cli/claudio.ts bin/claudio.cjs
"Take small steps. Do not automate slop. Fewer agents with better output beats MAXXING."
Nivel 1
Main Agent — Prototype Bench
You → Hermes Agent
  • Prototipando un workflow por primera vez
  • Testing de ideas antes de invertir en agente dedicado
  • Asistente personal (inbox, voice memos, calendario)
  • Campaign briefs únicos
Setup:
claudio hermes install
claudio hermes init programmer
↑ Subir al Nivel 2 cuando: el workflow se repite 3+ veces
Nivel 2
Agentes Especializados — Un Rol Cada Uno
You → hermes-seo
You → hermes-cmo
You → hermes-outbound
You → hermes-design / hermes-ops
  • Workflow repetible con pasos definidos
  • Scope sin solapamiento entre agentes
  • Credenciales + memoria independiente por agente
Setup:
mkdir -p ~/.hermes/agents/hermes-seo
claudio hermes init researcher
↑ Subir al Nivel 3 cuando: la tarea es cross-functional
Nivel 3
Equipo Orquestado — El Orquestador Enruta
You → Orchestrator
↓
Specialist Agents
  • Tarea toca más de un rol
  • El orquestador decide quién hace qué
  • Síntesis de resultados de múltiples agentes
SOUL del orquestador:
Scope de cada agente + protocolo de routing + regla de síntesis
↑ Subir al Nivel 4 cuando: output consistente, no necesitas estar en el loop
Nivel 4
Equipo Automatizado — Cron + Eventos
Cron / Events
↓
Orchestrator → Agent Team
↓
Reports (archivados)
  • Workflow probado en Nivel 2 y Nivel 3 primero
  • Reportes semanales de SEO, audits nocturnos
  • Event-driven: lead qualification, inbox triage
  • Usuario notificado solo cuando necesita criterio
Regla clave: No se automatiza slop. El output debe ser consistentemente bueno antes de automatizar.

🔍 Árbol de diagnóstico — ¿qué nivel necesito?

1. ¿Ya ejecuté este workflow antes?
   NO  → Nivel 1 (prototipa primero)
   SÍ  → continuar ↓

2. ¿El workflow toca más de un rol?
   NO  → Nivel 2 (agente especializado)
   SÍ  → continuar ↓

3. ¿Quiero estar en el loop para cada ejecución?
   SÍ  → Nivel 3 (orquestado, tú supervisas)
   NO  → continuar ↓

4. ¿El output del workflow es consistentemente bueno?
   NO  → volver al nivel anterior y mejorar primero
   SÍ  → Nivel 4 (automatizar)

📏 Las 5 Reglas del Proyecto

Constraints no negociables definidos en .claude/rules/. Aplicadas a cada agente en cada tarea.

security.md orchestration.md domain.md frontend.md html-first.md

🔒 security.md — Seguridad y OWASP

Datos sensibles: nunca en localStorage, nunca en logs de consola, nunca en client-side env vars. Todas las queries deben usar prepared statements. Sin dangerouslySetInnerHTML sin DOMPurify. API keys y tokens: solo server-side. Los hooks bloquean automáticamente las violaciones críticas.

🔄 orchestration.md — Pipeline de 7 fases

Todo feature, bugfix o refactor relevante pasa por las 7 fases obligatorias. Ninguna fase se saltea. Si un equipo devuelve BLOQUEADO, el pipeline se detiene y se corrige antes de avanzar. El estado del pipeline se registra en pipeline-state.json.

🏢 domain.md — Terminología de dominio

Define el vocabulario canónico del negocio. Solo se usan los términos del glosario en labels de UI, nombres de variables, comentarios y documentación. Los invariantes de negocio definidos aquí se convierten en tests obligatorios.

🎨 frontend.md — Convenciones de frontend

Design tokens CSS (nunca hardcodear valores), cn() para clases condicionales, componentes en directorios estándar, no any, iconos solo de lucide-react, useMemo/useCallback para objetos en render, keys de lista siempre con ID único.

📄 html-first.md — HTML sobre Markdown para outputs de agentes

Todo artefacto que un humano va a leer (reportes, specs, auditorías, diagramas, dashboards) se genera como HTML autocontenido, no Markdown. El HTML debe ser: fully self-contained (sin CDN externo), dark mode, con status badge en el header, y footer con timestamp. Guardar en docs/.
